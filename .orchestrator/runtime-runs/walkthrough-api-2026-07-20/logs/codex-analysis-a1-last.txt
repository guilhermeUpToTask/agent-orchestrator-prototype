I concur with both root causes.

1. The storm is caused by a recovery-boundary mismatch: `_reserve_goal_promotion()` raises the domain-level `TaskFailed` inside txn1, while the existing handler covers only the later runner side effect. The exception escapes `handle()`, the worker retries unchanged state, and the deterministic failure repeats indefinitely.

Your fix is directionally correct, but transaction safety depends on `_reserve_goal_promotion()` being validation-first. If it mutates the plan, reserves resources, emits events, or writes through the UoW before checking evidence, catching the exception and committing a `PlanBlock` in the same transaction could preserve partial promotion state. The robust options are:

- Ensure all promotion preconditions are checked before mutation.
- Or roll back txn1 and create the block in a fresh transaction.

Also consider concurrency/idempotency: repeated ticks must not create duplicate blocks.

2. Likely latent defects in the same family:

- Audit every operation called before the runner boundary—`peek_next`, legacy completion, promotion reservation, task start, and pause/review transitions—for `TaskFailed` or equivalent recoverable domain exceptions escaping txn1.
- Centralize recovery around the entire atomic dispatch/reservation phase, while distinguishing recoverable plan failures from infrastructure/programming errors.
- `peek_next()` and promotion eligibility currently encode inconsistent predicates. Returning `(goal, None)` means “promotable,” but apparently checks only terminal/nonfailed tasks, whereas promotion additionally requires every task to be `DONE` with evidence. A skipped/cancelled terminal task, or a `DONE` task lacking evidence, can therefore produce a permanently unpromotable navigation result. Blocking prevents the storm but does not resolve the semantic mismatch. Navigation should return a distinct blocked/invalid state, or use the same canonical `can_promote` predicate as reservation.
- Verify recovery actions can actually repair each condition. `retry_stage`, `edit_task`, and `start_replan` must allow missing evidence or non-`DONE` terminal states to be corrected; otherwise the block is recoverable only nominally.
- Review transaction exit behavior: if returning from the catch commits automatically, confirm the failed promotion’s transient events and reservations are absent.

3. The pause failure is phase/status model drift. The guard should use the cyclic lifecycle as the authority when a cycle exists, with legacy phase only for legacy plans.

Preserve graceful pause semantics by distinguishing `request_pause` from immediate `pause`: during an active runner invocation, set `pause_requested` and let finalization/checkpointing complete; do not transition to fully paused or invalidate the active claim prematurely. At the next dispatch boundary, the existing early check can return `PAUSED`. Ensure legal-actions calculation and mutation validation share exactly the same predicate, including active-run, terminal, blocked, and already-requested cases.
