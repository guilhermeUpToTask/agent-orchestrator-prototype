§Code review findings

Process-lifecycle observations are logged, not persisted cli_runner.py:250–260

The task asked for typed observations "via app/observations.py contracts," and the shape is correctly reused — but the wiring in _run_sync's on_observation callback only forwards each TelemetryObservation to structlog. Nothing calls the observation repository, so PROCESS_STARTED/PROCESS_EXITED/PROCESS_TIMED_OUT records never reach the durable, queryable ledger the reasoner's model.usage observations already use. It's diagnostic-log-only, not the canonical operational evidence Phase 2 of the architecture established.

on_observation=lambda observation: log.info(
    "runtime.process_observation", kind=observation.kind.value, ...
),

Not a regression — nothing worse than before — but it's the one place this task fell short of its own stated goal. Threading an ObservationRecorder through CliAgentRunner.run() alongside the existing event_sink parameter is the natural follow-up.

Constructor default builds a fresh AppContainer outside the composition root cli_runner.py:98–103

CliAgentRunner.__init__ falls back to AppContainer.from_env() (via a deferred import, to dodge a circular import with container.py) when no orchestrator_home is supplied. In the real wired path this is dead code — container.py always passes its own orchestrator_home down through factory.py's CatalogAgentRunner — but it technically reads the environment outside the one place CLAUDE.md's invariant #6 reserves for that. It exists purely as a test-construction convenience; a plain env lookup matching AppContainer's own default, rather than instantiating a whole container, would say the same thing without the smell.

Module docstring didn't get the memo cli_runner.py:1–16

The file header still reads "Streamed NDJSON events … are roadmap 2.4" and describes the subprocess as simply "blocking." Streaming has landed; the docstring is now the stale artifact CLAUDE.md's docs-discipline rule is written to catch — just one file down from where that rule usually applies.

In-memory output is unbounded even though the log file isn't process_supervisor.py:159–174

_BoundedLog correctly caps the on-disk JSONL file, but the outputs["stdout"]/["stderr"] lists accumulate every line for the life of the process before the final _OUTPUT_TAIL_CHARS truncation happens at the very end in cli_runner.py. A pathological command that writes gigabytes before hitting the wall-clock timeout can still grow memory unbounded. Pre-existing risk class under the old communicate() call too, so not a regression — just a missed chance to close it while the code was already being rewritten.

A real bug got caught and fixed mid-flight process_supervisor.py:64–83 (fixed by grok, task 4)

The original truncation logic could slice the retained log tail at an arbitrary byte offset, landing mid-JSONL-line and leaving an unparseable fragment at the start of the file. While writing tests for the truncation path, the second attempt on streaming-tests found this and rewrote it to drop the leading partial line before writing the marker. This is exactly the kind of thing a second pair of eyes — human or not — is for, and it's now covered by a dedicated test.

Endpoint security is correct by default plans.py — attempt_log

The new endpoint checks attempt.plan_id != plan_id before serving anything, returning the same 404 for "doesn't exist" and "belongs to a different plan" rather than leaking existence. It routes through the single existing error-mapping table (AttemptNotFoundError → ATTEMPT_NOT_FOUND → 404) instead of inventing a parallel error path, and reads the JSONL log defensively — a malformed or half-written line is skipped, not a 500.

§Recommendations

  1. 01Wire on_observation to the real observation repository, not just structlog — thread an ObservationRecorder through CliAgentRunner.run() the same way event_sink already flows.
  2. 02Bound outputs[stream] in supervise_process to match the on-disk log cap, so a runaway process can't grow memory unbounded before its timeout fires.
  3. 03Replace the AppContainer.from_env() constructor fallback in CliAgentRunner with a plain environment default, removing the deferred import and the composition-root smell.
  4. 04Refresh the cli_runner.py module docstring — streaming has shipped, the "blocking subprocess" description is stale.
  5. 05Don't route non-trivial write tasks to pi_free (nemotron-nano free tier) until root-caused — it returned a clean exit with zero output on two unrelated real tasks this run, with no error signal to act on.
  6. 06Give mimo's native free model one more trial now that the wrapper actually records timeout evidence — this run's one data point is inconclusive, not damning.
